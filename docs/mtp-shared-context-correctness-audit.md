# MTP Shared Context Correctness Audit

This audit is for the current experimental diff at `bea9b4ffa6bd`. It focuses
on correctness traps that matter before any PR split. It does not certify the
branch as production-ready.

## Audit table

| Risk | Relevant code | Why risky | Current mitigation | Test that should catch it | Remaining uncertainty |
| --- | --- | --- | --- | --- | --- |
| Cross-slot KV contamination | `examples/server/server-context.cpp`: `server_context::ctx_mtp_shared`, `companion_ctx_for_slot()`, `clear_slot_companion_state()`, batched draft path around `common_speculative_draft_mtp_batch()` | One companion `llama_context` holds KV for multiple slots. A missed `slot.id`, stale KV range, or full-context clear at the wrong time can leak one slot's draft state into another. | Companion operations use the slot's sequence id. Release/reuse paths call `clear_slot_companion_state()`. Batched draft input carries a per-slot `seq_id`. Shared path is env-gated. | Two concurrent requests with visibly different prompts, followed by same-server slot reuse. Check for stale continuations, mixed-sequence logs, invalid logits, and non-200 responses. | Needs broader coverage for every target KV mutation path, especially prompt-cache and context-shift paths under load. |
| Sequence ID remapping | `common/speculative.cpp`: `common_speculative_mtp_draft_input`, `common_speculative_draft_mtp_batch()`; `examples/server/server-context.cpp`: batched draft input assembly | Batched decode must map each row back to the correct slot, sequence id, base position, hidden row, and result vector. One off-by-one can make tokens or hidden state belong to the wrong slot. | The batched helper accepts explicit `seq_id` and `draft_base_pos`; it returns `false` for unsupported shapes so callers can fall back to the per-slot path. | Concurrent prompts with different accepted/rejected draft lengths. Verify accept rates per slot, output sanity, and no mixed-sequence warnings. | Hidden-state row ordering is still subtle and should be reviewed carefully before batched draft is proposed. |
| Repeated sequence IDs in validation batches | `examples/server/server-context.cpp`: `server_can_defer_speculative_checkpoints_to_ranges()`, `server_find_single_speculative_slot_in_range()`, `speculative_decoding_accept(int32_t batch_start, int32_t batch_end)` | Continuous batching can put several tokens for one slot or interleaved tokens for several slots in the same decode batch. Saving one checkpoint for the wrong mixed range corrupts restore semantics. | Range checkpointing only saves when the batch range maps to exactly one speculative slot. The accept path can be called over a range instead of the whole batch. | Force `-np 2` with uneven prompt and generation lengths so ranges interleave. Check range checkpoint log lines and accept correctness. | More pathological scheduler interleavings could still be under-tested. |
| Slot release cleanup | `examples/server/server-context.cpp`: `release_slot()`, `clear_slot_companion_state()`, cancellation/release call sites | A released slot can be immediately reused. If companion KV or hidden validity survives release, the next request can inherit stale draft state. | `release_slot()` centralizes release and calls `clear_slot_companion_state(slot, true, reason)`. Final response, cancellation, empty prompt, embedding completion, decode failure, and system prompt reset routes call it. | Same-server slot reuse after successful generation and after cancellation. Inspect logs for release reason and companion clear. | Direct future writes to `slot.release()` could bypass `release_slot()`. Keep review focused on preventing new bypasses. |
| Destructor order | `examples/server/server-context.cpp`: `server_context::~server_context()` | `common_speculative` objects may hold a borrowed pointer to the shared companion context. Freeing the shared context before freeing slot speculative state would leave dangling references. | Destructor frees each `slot.spec` first, then frees `ctx_mtp_shared`, then frees draft model/context state and target model/context. | Start shared-MTP server, run a short request, then stop cleanly. Run under sanitizer or debug build if practical. | This assumes `common_speculative_free()` never reaches back into a context already freed elsewhere. Review ownership comments before final PR. |
| Target-only prompt-cache restore | `examples/server/server-context.cpp`: prompt cache load path around `clear_slot_companion_state(*ret, true, "target-only prompt cache restore")` | Prompt cache restore mutates target context only. Companion context could retain old draft KV and hidden state at positions that no longer match target. | Target-only prompt cache restore clears companion state. | Enable prompt cache, run a reusable prompt, restart or reload the cache, then generate with shared MTP. Check output sanity and clear logs. | Needs a deterministic prompt-cache regression test, not only manual smoke. |
| Slot state restore/reset | `examples/server/server-context.cpp`: slot state restore failure/success paths and slot erase paths | Restoring or erasing target slot state without clearing companion state creates divergent target and companion contexts. | Restore failure clears companion state. Target-only slot state restore clears companion state. Slot erase clears companion state. | Save/restore a slot state, then continue generation under shared MTP. Also test failed restore with invalid file. | File-based restore paths are less frequently exercised than normal completion. |
| Target context checkpoint restore/reset | `examples/server/server-context.cpp`: `apply_checkpoint()`, context checkpoint restore/reset branches | Server prompt checkpoints operate on target context. If companion state survives a target checkpoint restore or reset, speculative decode starts from a false prefix. | Restore and reset branches call `clear_slot_companion_state(slot, true, ...)`. | Prompt-cache or long-prompt shape that triggers `apply_checkpoint()`. Check the target checkpoint restore/reset logs and next generation. | Dense restore/redecode cost can mask subtle correctness issues; output sanity matters too. |
| Checkpoint save/restore scope | `examples/server/server-context.cpp`: `save_speculative_checkpoint()`, `restore_speculative_checkpoint()`, process-range checkpoint save, fallback restore | Recurrent checkpoint state must correspond to the exact slot, prefix position, and mode. A checkpoint saved before the wrong range or restored after wrong acceptance corrupts recurrent state. | Slot-local `server_speculative_checkpoint` tracks mode and `n_past`; range checkpoint save is gated by isolated slot range; restore discards checkpoint after use. | Rejection-heavy prompts with `--draft-p-min` low enough to trigger restore. Collect `ckpt_save`, `restore`, `fallback_call`, and `redecode` buckets. | Current shared parallel path avoids per-step; a future split-state per-step path needs a fresh audit. |
| Prompt prefill serialization | `examples/server/server-context.cpp`: prompt processing and batched draft scheduling | Shared companion context cannot safely accept unrelated prompt-prefill work as if it were per-slot local state. Concurrent prompts can reach speculative decode at different times. | Shared companion draft is only used under narrow env gates. Batched helper declines unsupported input shapes. Prompt reuse and trim paths clear or trim companion state. | Two long prompts with different lengths under `-np 2`, followed by generation and slot reuse. | Needs explicit tests for long prompt prefill plus early generation in another slot. |
| Context shift and prompt trim | `examples/server/server-context.cpp`: `shift_slot_companion_state()`, `trim_slot_companion_state()`, context shift, prompt reuse trim/full clear, string-ban rewind | Target KV can be shifted, trimmed, or rewound without a full request release. Companion state must be shifted the same way or cleared. | Dedicated helper calls were added for target context shift, prompt reuse trim/full clear, and string-ban rewind. | Force small-context shift, prompt reuse trim, and string-ban rewind under shared MTP. Check output sanity and helper logs. | Context-shift paths are complex and should stay in the required test list for lifecycle PRs. |
| KV clear scope | `examples/server/server-context.cpp`: `kv_cache_clear()` and target KV clear branch | Clearing target KV without clearing the shared companion context leaves all companion sequences stale. Clearing the whole companion context at the wrong time can affect other active slots. | Target KV clear calls companion clear per slot; if clearing a shared companion context globally, code must be careful about active slots. | Run multi-slot request, trigger KV clear/reset, then run reuse smoke. | Needs review for whether all active shared sequences are handled safely during global clear. |
| Fallback mode selection | `examples/server/server-context.cpp`: init-time checkpoint mode selection around shared MTP and range gate; `common/common.cpp`: `--recurrent-ckpt-mode` parsing | `per-step` is fast but unsafe in current shared parallel mode; CPU is safe-looking but very slow. Picking either silently creates bad behavior. | Range-gated shared parallel MTP forces `gpu-fallback`. The code logs warnings when forcing fallback. Single-slot auto remains `per-step`. | Verify logs for single-slot auto (`per-step`) and shared/range parallel (`gpu-fallback`). Compare CPU/gpu-fallback/per-step scouts. | Non-range shared parallel fallback policy needs clear maintainer review, because CPU fallback is expensive and per-step is unsafe. |
| CUDA per-step unsafe path | `examples/server/server-context.cpp`: comments around context-global per-step buffers; `src/llama.cpp`: recurrent checkpoint buffer implementation | Existing GPU per-step checkpoint buffers are context-global. Shared parallel slots can race or overwrite state. | Shared parallel range checkpoint avoids per-step and uses `gpu-fallback`. Failed scout is documented. | Optional known-failing control: dense Qwen3.6 `-np 2` shared parallel per-step. Expected failure signature is CUDA `invalid argument`; do not run as a normal pass test. | A real fix requires split-state per-step checkpoint buffers, not only range isolation. |
| Flash-attention Qwen3Next crash | Dense Qwen3.6 no-MTP FA control; crash in `iqk_fa_templates.h` | The dense parallel-slot FA crash is separate from shared MTP. Mixing it into shared-MTP tests will create false failures. | Shared-MTP evidence uses `--no-flash-attn`. Test matrix keeps FA as a known-failing control. | Run only explicit FA control if needed: dense `-np 2`, no MTP, FA enabled. | Needs a separate issue/fix outside this PR split. |
| Env-gate interactions | `examples/server/server-context.cpp`: `server_mtp_shared_context_enabled()`, `server_mtp_batched_draft_enabled()`, `server_mtp_range_ckpt_enabled()` | Different combinations of shared context, batched draft, range checkpoint, and profile env can create untested shapes. | Gates are independent and the batched helper has a fallback path. Evidence is limited to specific combinations. | Matrix runs with shared only, shared+batch, shared+batch+range, profile off, profile on, and single-slot controls. | Combinatorial coverage is still incomplete. Final PRs should reduce or remove temporary gates as behavior stabilizes. |
| Graph hidden-state row shape | `src/graphs/build_glm4.cpp`, `src/graphs/build_qwen35.cpp` | Batched MTP draft with `n_tokens > 1` needs a 2D hidden-state tensor. Wrong shape can corrupt hidden-state rows or crash graph build. | Graph builders use 2D hidden-state tensors when `n_tokens > 1`. | Batched draft smokes on dense and MoE with `spec_draft_batched` nonzero. Add a debug assert if practical. | Other model graph builders are not covered by these Qwen/GLM changes. Keep batched draft model scope narrow. |
| External draft-model behavior | `examples/server/server-context.cpp`: shared context initialization checks `params_base.speculative.model_dft == nullptr`; `common/speculative_init()` accepts optional shared context | Accidentally applying shared self-MTP context ownership to external draft models could regress a different feature. | Shared context is created only for self-MTP shape with no external draft model. External draft sharing is documented out of scope. | External draft smoke with shared env set should either use existing behavior or cleanly decline shared self-MTP path. | External draft context sharing needs its own design if maintainers want it. |
| Hidden-state restore after rejection | `examples/server/server-context.cpp`: `restore_speculative_checkpoint()`, accept path around rejection restore; `common/speculative.cpp`: MTP hidden feature extraction | Rejection restore must leave target hidden state, MTP hidden rows, sampler state, and output rows consistent. Dense restore/redecode cost is already large, so subtle correctness bugs can hide behind slow paths. | Per-step restore commits hidden from accepted-prefix rows; fallback restore re-decodes accepted tokens and commits output/hidden state. Profile buckets expose restore/redecode cost. | Rejection-heavy prompts, low draft acceptance threshold, and deterministic output comparison across repeated runs. | Dense Qwen3.6 still spends heavy time in restore/redecode. That is a performance problem and a correctness area worth targeted review. |

## Human review priorities

1. Verify every target-context mutation either mirrors into companion state or
   invalidates companion state.
2. Review batched draft row ordering and sequence-id mapping before showing that
   shard.
3. Keep single-slot MTP `auto -> per-step` behavior intact.
4. Keep shared parallel `per-step` disabled until checkpoint buffers are truly
   per-slot.
5. Keep dense flash-attention failure out of the shared-MTP proof set.
